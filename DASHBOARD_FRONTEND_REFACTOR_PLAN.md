# Aneiang.Yarp.Dashboard 前端优化及 BUG 修复方案

## 1. 背景与目标

`Aneiang.Yarp.Dashboard` 的前端目前基于 Razor 视图 + 原生 JavaScript + Bootstrap 实现，具备基础功能，但如果希望它真正成为一个适合开发、运维、平台人员长期使用的 Dashboard，页面层仍然需要一次系统性的重构。

这份文档的目标不是“在现有代码上修修补补”，而是给出一份**基于原生 + Bootstrap 的分阶段重构实施方案**，让前端最终达到：

- 结构清晰
- 状态稳定
- 交互统一
- 配置编辑可靠
- JSON 模式与 schema 驱动一致
- 导入 / 导出标准 YARP 配置可用
- 新增 / 修改集群和路由真正可用
- 日志排障体验明显提升
- 不依赖 CDN，可本地独立部署

本方案只使用：
- Razor 视图
- 原生 JavaScript
- Bootstrap
- 本地可部署的 JSON 编辑器 / 代码编辑器 / 语法高亮库（如需要）

不引入 Preact / React / Vue 架构。

---

## 2. 明确约束

### 2.1 不使用 CDN

前端所有资源必须本地化，不能运行时依赖任何外部 CDN。

#### 约束要求
- 所有 JS / CSS / 字体 / 图标 / 编辑器资源都必须本地托管
- 页面启动不能依赖外网资源
- 生产、内网、离线环境都要可用
- 发布物中不能包含运行时 CDN 引用

#### 落地原则
- 可以在构建阶段通过 npm / pnpm / yarn 安装依赖
- 最终产物必须是本地静态文件
- 运行时只能引用项目内静态资源
- 如果使用第三方编辑器、图标或语法高亮库，必须以本地方式打包进项目

---

### 2.2 JSON 编辑器可以重新选型
JSON 编辑器不局限于当前实现，只要满足本方案要求即可。

#### 选型目标
- 支持大文本编辑
- 支持 JSON Schema 驱动的提示
- 支持字段补全、枚举提示、默认值提示、层级提示
- 支持格式化、折叠、跳转、错误定位
- 支持与表单模式共享同一份数据模型
- 支持导入 / 导出标准 YARP 配置
- 支持纯本地部署

#### 可选实现方向
- Monaco Editor：适合 IDE 风格体验，JSON Schema 能力强
- CodeMirror 6：轻量、可扩展、适合嵌入 Dashboard
- Ace Editor：成熟、易用、体积较轻
- 其它可本地部署方案：只要满足 schema 提示和离线部署要求即可

#### 选型原则
- 不使用 CDN
- 可本地构建与发布
- 必须能根据 `ConfigurationSchema.json` 提示和校验
- 必须支持标准 YARP JSON 的导入与导出

---

### 2.3 JSON 模式必须与 `ConfigurationSchema.json` 保持一致
这是一条强约束。

#### 要求
- JSON 模式新增、修改、导入、导出时的字段结构、层级、枚举、默认值、类型约束，必须与 `src/Aneiang.Yarp.Dashboard/wwwroot/ConfigurationSchema.json` 保持一致
- 用户在 JSON 模式看到的体验，应尽量接近“直接编辑配置文件”
- 用户应该感觉自己编辑的是一份标准、可导入、可导出的 YARP 配置文件，而不是 Dashboard 自定义格式

#### 具体目标
- JSON 编辑区域必须能识别 schema 中的字段提示
- JSON 结构必须可与表单模式互相转换
- 导出结果应尽可能是标准 YARP 风格配置
- 导入结果应能识别标准 YARP 配置并映射回编辑模型
- 不允许前端私自维护一套和 schema 无关的隐藏格式

---

## 3. 当前前端常见问题类型

结合 Dashboard 的页面形态，前端问题通常集中在以下几类：

### 3.1 页面结构混杂
- 页面布局、状态、渲染、API 调用混在一个脚本里
- 多页面共享逻辑重复实现
- 公共工具散落在不同 JS 文件中

### 3.2 状态管理混乱
- tab、筛选、展开、loading、分页等状态分散
- 页面切换后状态丢失或不一致
- 通过全局变量互相传值，容易冲突

### 3.3 数据渲染不稳定
- 后端字段变化后前端无法兼容
- 列表与详情使用不同数据组织方式
- 某些字段为空时页面布局错位

### 3.4 交互体验不足
- 搜索、筛选、复制、展开操作不一致
- 日志页面信息密度高但层级不清楚
- 错误状态、空状态、加载状态不够统一

### 3.5 潜在 BUG
- DOM 节点不存在时脚本报错
- 组件初始化顺序依赖过强
- 按钮点击后状态不回收
- 语言切换、登录态、刷新逻辑存在同步问题
- tab 之间刷新和初始渲染可能互相影响

### 3.6 新增 / 修改集群和路由存在体验和正确性问题
这是当前最值得优先重构的痛点之一：
- 表单字段太多，用户不知道从哪里开始
- JSON 模式和表单模式之间不统一
- 修改时不容易看清楚前后差异
- 保存后不知道是否真的生效
- 缺少回滚、预览和验证流程

## 4. 前端重构原则

### 4.1 独立于现有实现重做
允许不参考现有脚本内部结构，直接重构为新的前端组织方式。

### 4.2 分层明确
前端拆分为：
- 基础设施层
- 状态层
- 渲染层
- 页面模块层
- 交互层
- 弹窗层
- 编辑器层
- 样式层

### 4.3 以 Dashboard 场景为中心
Dashboard 是运维工具，不是营销型页面。优先目标应是：
- 快速定位问题
- 快速查看配置
- 快速筛选日志
- 快速复制内容
- 快速排障
- 快速新增与修改配置

### 4.4 优先稳定，再追求美观
先修复数据、状态、交互、兼容性问题，再优化视觉。

### 4.5 所有页面组件可复用
例如：
- 搜索框
- Badge
- 折叠面板
- 空状态
- 错误状态
- 复制按钮
- 高亮标签
- JSON 编辑器容器
- 差异预览面板
- 校验结果面板

## 6. 页面级重构目标

## 6.1 总览页（Home / Index）

### 目标
首页要成为 Dashboard 的“总指挥台”，展示整体状态，而不是单纯堆叠信息块。

### 建议保留内容
- 网关版本
- 环境名
- 启动时间
- uptime
- 内存使用
- 主机名
- cluster 数量
- route 数量
- 日志开关状态
- 最近活动摘要
- 当前登录/鉴权状态
- 当前语言

### 优化方向
1. **首屏分区清晰**
   - 顶部：状态总览
   - 中部：clusters / routes / logs 三个核心入口
   - 底部：最近事件或快捷操作

2. **减少视觉噪音**
   - 避免所有信息都平铺在同一层
   - 关键指标使用卡片化展示
   - 次要信息折叠或收纳

3. **增强可操作性**
   - 提供刷新按钮
   - 提供快速跳转到 cluster / route / log
   - 提供快速进入新增 / 修改配置入口
   - 提供当前登录态提示

### 典型 BUG 重点
- 页面初始加载时脚本顺序错误导致统计不显示
- 某些统计字段为空时布局抖动
- 切换语言后统计标题未同步
- Tab 初次渲染和刷新冲突

---

## 6.2 集群页（Clusters）

### 目标
让用户快速理解每个 cluster 的健康度、下游、可编辑性和关键配置，并且支持新增、修改、导入、导出、回滚。

### 页面结构建议
每个 cluster 使用一张独立卡片，卡片分四段：

#### A. 摘要区
- clusterId
- 是否可编辑
- 来源（静态 / 动态）
- 健康状态概览
- destination 总数
- healthy / unknown / unhealthy 统计
- 最后更新时间

#### B. 配置区
- load balancing policy
- session affinity
- health check
- http client
- http request

#### C. 详情区
- destinations 列表
- metadata
- cookie 配置
- 高级字段

#### D. 操作区
- 新增
- 修改
- 导入
- 导出
- 复制 JSON
- 回滚
- 删除（如权限允许）

### 交互建议
- 默认只显示摘要
- 点击展开配置
- 展开后再显示 destinations
- 提供复制按钮
- 支持 clusterId 搜索
- 支持按健康状态过滤
- 支持按 editable 状态过滤
- 支持按来源过滤
- 支持新建 / 编辑 / 导入 / 导出 / 回滚流程

### 典型 BUG 重点
- destination 列表多行时表格错位
- 详情区展开后没正确回收高度
- 某些配置为空时对象访问报错
- status badge 与实际健康状态不一致
- 编辑后保存成功但列表未刷新
- 导出结果与实际配置结构不一致

---

## 6.3 路由页（Routes）

### 目标
让路由规则能够被快速定位、快速比较、快速复制、快速修改，并且支持标准 JSON 模式编辑。

### 页面结构建议
每个 route 使用卡片或折叠行展示：

#### A. 摘要区
- routeId
- path
- clusterId
- order
- source（config / dynamic）
- editable
- 状态标签

#### B. 匹配规则区
- methods
- hosts
- headers
- queryParameters

#### C. 规则详情区
- transforms
- authorizationPolicy
- corsPolicy
- outputCachePolicy
- rateLimiterPolicy
- timeoutPolicy
- timeout
- metadata

#### D. 操作区
- 新增
- 修改
- 导入
- 导出
- 复制 JSON
- 回滚
- 删除（如权限允许）

### 交互建议
- routeId / clusterId 联合搜索
- 对 transforms 用步骤式显示
- 对 headers / queryParameters 用 badge 显示
- 高亮路径和方法
- 一键复制 route JSON 或关键字段
- 支持 JSON 模式和表单模式双编辑

### 典型 BUG 重点
- transforms 结构变化后渲染异常
- methods / hosts 为空导致空白显示问题
- route 排序后 UI 与数据顺序不一致
- 长 path 导致布局撑开
- 修改后 JSON 模式和表单模式不同步
- 导出为标准配置时字段丢失

---

## 6.4 日志页（Logs）

### 目标
将日志页从“纯列表”升级为“排障工作台”。

### 页面结构建议

#### 顶部工具栏
- 日志开关状态
- 自动刷新开关
- 条数选择
- route 筛选
- status 筛选
- traceId 搜索
- 清空日志按钮
- 导出日志按钮（如需要）

#### 中部列表区
每条日志只显示摘要：
- 时间
- level
- category
- message
- routeId
- traceId
- status code

#### 详情区
点击后展开：
- request body
- response body
- headers
- downstream url
- exception
- 原始事件信息
- 复制按钮

### 交互建议
- request / response 成对显示
- 同一 trace 聚合
- body 懒加载展开
- 长文本截断
- 支持复制整条事件
- 支持按状态码颜色区分
- 支持清空日志二次确认
- 支持日志滚动定位

### 典型 BUG 重点
- 日志滚动容器和页面滚动冲突
- 自动刷新导致展开项丢失
- 大 body 渲染卡顿
- 清空日志后 UI 未刷新
- 监听按钮状态和实际状态不同步
- 导出日志与页面显示不一致

---

## 6.5 配置编辑页 / JSON 模式编辑页

### 目标
把新增、修改、导入、导出、预览、校验统一成一个配置编辑工作台。

### 核心理念
- 表单模式用于普通用户
- JSON 模式用于高级用户
- 两种模式必须共享同一份数据模型
- JSON 模式必须与 `ConfigurationSchema.json` 保持一致
- 导入 / 导出必须支持标准 YARP 配置

### 页面结构建议

#### 左侧编辑区
- 表单模式
- JSON 模式
- schema 提示区
- 导入按钮
- 导出按钮

#### 右侧辅助区
- 实时校验结果
- 当前 schema 提示
- 预览 JSON
- diff 面板
- 变更摘要
- 版本信息

### 交互建议
- 表单和 JSON 模式切换时保持同一份草稿
- 输入 JSON 时自动联想字段、枚举、默认值、层级
- 提供错误定位到具体 path
- 支持一键格式化
- 支持一键导入标准 YARP 配置
- 支持一键导出标准 YARP 配置
- 支持保存草稿
- 支持版本比较与回滚

### 典型 BUG 重点
- JSON 和表单双向同步失败
- schema 更新后提示不一致
- 导入标准配置后字段映射丢失
- 导出结果不是标准配置
- 复杂对象折叠状态丢失
- 大文件编辑卡顿

## 7. 页面基础设施重构方案

## 7.1 API 封装层

### 目标
所有接口请求统一处理。

### 建议能力
- 自动附带 token
- 统一 JSON 解析
- 统一错误处理
- 统一超时处理
- 统一 HTTP 状态码处理
- 统一登录失效跳转
- 统一导入导出文件请求

### 常见 BUG 修复点
- 某些 fetch 没加认证头
- 非 200 返回未处理
- 403 / 401 响应导致页面静默失败
- 下载类请求被误当成 JSON 请求

---

## 7.2 状态管理层

### 目标
让页面状态集中管理，不再依赖零散全局变量。

### 建议状态项
- currentTab
- currentLocale
- currentToken
- clustersFilter
- routesFilter
- logsFilter
- logsCount
- expandedIds
- autoRefreshEnabled
- lastRefreshTime
- loadingFlags
- editorMode
- draftConfig
- activeSchemaPath
- importPreview
- exportPreview

### 推荐能力
- 统一读写
- 订阅状态变化
- 切页保留状态
- 刷新后恢复视图
- 编辑草稿持久化到 localStorage

---

## 7.3 渲染层

### 目标
所有页面的 DOM 渲染逻辑集中化。

### 建议组件
- `renderCard`
- `renderBadge`
- `renderEmptyState`
- `renderErrorState`
- `renderLoadingState`
- `renderKeyValueList`
- `renderJsonViewer`
- `renderCollapsibleSection`
- `renderCopyAction`
- `renderDiffPanel`
- `renderValidationPanel`

### 关键原则
- 渲染函数纯粹
- 不读接口、不改状态
- 输入数据 -> 输出 HTML
- 大文本使用懒渲染

---

## 7.4 弹窗层

### 目标
所有弹窗统一封装，避免页面内重复写确认框。

### 建议弹窗类型
- 登录弹窗
- 清空日志确认弹窗
- 复制成功提示
- 错误提示
- 配置详情弹窗
- 原始 JSON 弹窗
- 新增/修改确认弹窗
- 删除确认弹窗
- 回滚确认弹窗
- 导入确认弹窗

---

## 7.5 编辑器层

### 目标
让 JSON 编辑器、schema 校验、格式化、自动补全、导入导出形成独立能力。

### 建议职责
- JSON Schema 读取
- 代码提示
- 枚举提示
- 默认值提示
- 折叠与展开
- 格式化
- 诊断错误
- 导入标准配置
- 导出标准配置

### 关键原则
- 编辑器实现可替换
- UI 不绑定某一款编辑器
- schema 服务与编辑器适配层分离

---

## 8. 视觉与布局优化方案

### 8.1 视觉方向
Dashboard 建议采用“运维面板风格”，重点是：
- 清晰
- 层级明确
- 信息密度高但不乱
- 适合长时间盯屏排障
- 适合做配置修改和导入导出

### 8.2 统一设计规范
- 卡片统一边距和圆角
- 标题统一字号层级
- 状态统一颜色语义
- 按钮统一主次级别
- 大 JSON / 长文本统一折叠显示
- 导入导出按钮样式统一
- 保存 / 回滚 / 校验操作有明确主次区分

### 8.3 响应式建议
- 大屏：三列或双列
- 中屏：双列
- 小屏：单列
- 表格在窄屏下自动变为卡片列表或横向滚动
- 编辑器在窄屏下折叠为上下布局

---

## 9. BUG 修复总清单

以下问题应作为前端重构时的重点排查项。

### 9.1 初始化类 BUG
- 脚本加载顺序不正确
- DOM 未就绪就执行渲染
- 某些全局对象未初始化
- 页面某些 panel 未注册却被访问

### 9.2 数据类 BUG
- 后端字段变更后前端仍按旧字段取值
- 空对象 / 空数组未做保护
- 字段类型不一致导致渲染失败
- request / response 数据结构不一致
- 导入导出结构不一致

### 9.3 交互类 BUG
- 切 tab 后状态丢失
- 展开后再刷新丢失定位
- 自动刷新与手动刷新冲突
- 复制按钮重复绑定事件
- 清空日志后未重置 UI
- JSON 模式和表单模式切换不同步

### 9.4 性能类 BUG
- 大量日志一次性渲染导致卡顿
- 大 JSON 直接 innerHTML 导致页面迟缓
- 多次重复绑定事件
- 频繁 DOM 重绘未节流
- 编辑器在大配置下明显迟钝

### 9.5 兼容类 BUG
- 浏览器对某些 JS 语法兼容问题
- 特定宽度下表格布局错乱
- 英文文案长度导致按钮溢出
- 主题或字体变化后排版问题

### 9.6 配置编辑类 BUG
- JSON 输入没有 schema 提示
- JSON 校验不准确
- 导入后字段映射错误
- 导出不是标准 YARP 配置
- 保存后预览与实际不一致
- 版本回滚后页面未刷新

## 11. 推荐的技术实现方式

前端实现方式不限制于现有写法，允许采用“无 CDN、本地可部署、可渐进重构”的方案。

### 方案 1：继续原生 JS + Bootstrap
适合当前项目，改造成本低。

优点：
- 无额外运行时依赖
- 与 Razor 页面集成自然
- 易渐进式替换
- 最容易满足离线与内网部署要求

缺点：
- 状态管理和复杂交互仍要自己维护
- JSON 编辑器能力需要额外集成

### 方案 2：局部引入本地编辑器能力
例如 Monaco、CodeMirror 6、Ace 等，只作为局部编辑器能力引入。

优点：
- 可显著提升 JSON 编辑体验
- schema 提示能力更强
- 适合复杂配置编辑器

缺点：
- 需要构建链
- 需保证本地化资源完整
- 需要额外处理编辑器与表单的数据同步

### 推荐结论
本项目前端主框架建议保持 **原生 JS + Bootstrap**。
JSON 编辑器可以局部引入一个本地部署的成熟编辑器，但不引入 Preact / React / Vue 架构。
