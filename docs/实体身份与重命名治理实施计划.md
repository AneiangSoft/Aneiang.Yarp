# 实体身份与重命名治理实施计划

> 目标：解决 `routeId` / `clusterId` / `policyId` 被自由修改后导致策略、熔断、日志、审计、通知、配置历史等关联错乱的问题。

## 核心原则

- 当前配置关系使用不可变内部 UID。
- YARP 对外配置继续使用可变业务 Key。
- UI 展示使用 DisplayName。
- 历史日志保留 Key Snapshot，不随当前 Key 重命名而变化。
- 所有 Key 修改必须走统一重命名服务，禁止普通编辑直接修改。

---

## M1：止血版（已开始执行）

目标：立即阻止继续产生新的错乱。

### 任务

- [x] 前端普通编辑锁定 `Route ID`。
- [x] 前端普通编辑锁定 `Cluster ID`。
- [x] 前端保存时拒绝 ID 变化。
- [x] 后端 `SaveRoute` 拒绝 body 中传入不同 `routeId`。
- [x] 后端 `SaveCluster` 拒绝 body 中传入不同 `clusterId`。
- [x] 增加独立“重命名”按钮和弹窗。
- [x] 增强现有 rename API，至少同步策略绑定和运行时状态。

### 验收

- 普通编辑无法修改 route/cluster ID。
- 直接调用保存 API 传不同 ID 返回 400。
- 现有重命名 API 仍可用。

## M2：统一重命名服务

状态：已完成过渡版。已新增 `IGatewayIdentityService` / `GatewayIdentityService`，并将 cluster/route rename 主流程、配置快照、策略绑定同步、熔断状态迁移集中到该服务。后续 UID 阶段继续扩展为真正的事务化身份服务。

目标：所有 Key 修改进入同一个事务入口。

### 新增服务

```csharp
public interface IGatewayIdentityService
{
    Task<RenameResult> RenameClusterAsync(string oldClusterKey, string newClusterKey, string? reason, string? operatorName);
    Task<RenameResult> RenameRouteAsync(string oldRouteKey, string newRouteKey, string? reason, string? operatorName);
}
```

### Cluster Rename 必须处理

- 校验新 Key 合法与唯一。
- 保存配置快照。
- 更新 cluster key。
- 更新 routes 中 cluster 引用（过渡期仍是 key）。
- 更新 cluster policy `applied_targets`（过渡期 JSON 数组）。
- 重建 YARP 内存配置。
- rekey 熔断运行时状态：`old:any` -> `new:any`。
- 写审计日志。
- 推送配置变更事件。

### Route Rename 必须处理

- 校验新 Key 合法与唯一。
- 保存配置快照。
- 更新 route key。
- 更新 route policy `applied_targets`（过渡期 JSON 数组）。
- 重建 YARP 内存配置。
- 写审计日志。
- 推送配置变更事件。

---

## M3：UID 双写

状态：已完成过渡版。已为 route/cluster/policy 实体增加 UID 字段；SQLite 自动迁移 `route_uid` / `cluster_uid` / `policy_uid`，旧数据自动回填；保存时双写 UID + 旧 Key；动态模型增加 `RouteUid` / `ClusterUid`，重命名时保留 UID 不变。

目标：数据库开始具备长期身份模型，同时兼容旧字段。

### 数据库新增字段

```sql
ALTER TABLE clusters ADD COLUMN cluster_uid TEXT;
ALTER TABLE routes ADD COLUMN route_uid TEXT;
ALTER TABLE policies ADD COLUMN policy_uid TEXT;
```

### 迁移

- 旧数据自动生成 UID。
- 新数据双写 UID + Key。
- 原 `cluster_id` / `route_id` / `policy_id` 暂时作为 Key。

### 验收

- 所有旧数据都有 UID。
- 新增数据有 UID。
- 旧 API 不受影响。

---

## M4：策略绑定关联表

状态：已完成过渡版。已新增 `policy_targets` 关联表和 `PolicyTargetEntity`；SQLite 自动创建并从旧 `applied_targets` JSON 迁移；策略服务 Apply/Unapply/Delete/Get 已切换到关联表，同时保留旧 JSON 字段兼容写入；重命名时同步 `target_key_snapshot`。

目标：策略绑定不再依赖可变字符串 Key。

### 新表

```sql
CREATE TABLE policy_targets (
    id TEXT PRIMARY KEY,
    policy_uid TEXT NOT NULL,
    target_type TEXT NOT NULL,
    target_uid TEXT NOT NULL,
    target_key_snapshot TEXT NOT NULL,
    created_at TEXT NOT NULL,
    UNIQUE(policy_uid, target_type, target_uid)
);
```

### 迁移

- 从旧 `policies.applied_targets` JSON 数组迁移。
- 根据 target key 查 route/cluster UID。
- 读写切换到 `policy_targets`。
- 旧字段保留一版兼容读取。

---

## M5：路由引用集群改 UID

状态：已完成过渡版。`DynamicRouteConfig` / `RouteEntity` 已增加 `ClusterUid`；SQLite `yarp_routes` 自动迁移 `cluster_uid` 并从 `yarp_clusters.cluster_uid` 回填；路由保存/加载双写 `cluster_uid + cluster_id`；内存路由在新增、更新、重命名和批量替换时同步 `ClusterUid`，YARP 输出仍使用 `cluster_id` 作为兼容 Key。

目标：核心配置内部引用稳定。

### 变更

- `routes` 增加/使用 `cluster_uid`。
- 内部 `DynamicRouteConfig` 引用 cluster uid。
- YARP 生成时把 `cluster_uid` 映射成 `cluster_key`。

### 验收

- cluster key 重命名不需要批量更新 routes。
- 生成出的 YARP `RouteConfig.ClusterId` 仍是 cluster key。

---

## M6：历史与运行时状态完善

状态：已完成过渡版。已为审计日志、代理日志、通知历史增加 UID 与 Key Snapshot 字段；SQLite 自动迁移并用旧 key 回填 snapshot；审计领域模型增加 target UID/snapshot 字段；代理日志和通知历史保存/读取支持 UID + snapshot。运行时熔断状态已在前序阶段支持 rename rekey，后续可继续切到 UID key。

目标：历史事实和当前关联同时成立。

### 日志/审计/通知增加

- `target_uid`
- `target_key_snapshot`
- `target_display_name_snapshot`

### 熔断状态长期改造

- 状态 key 从 `clusterKey:destinationKey` 改为 `clusterUid:destinationUid`。
- 展示时附带 key snapshot。
- 默认不持久化运行时状态；可配置开启。

---

## 推荐执行顺序

1. M1 止血：禁止普通编辑改 ID。
2. M2 统一重命名服务。
3. M3 UID 双写。
4. M4 策略绑定表。
5. M5 路由引用 cluster uid。
6. M6 历史和运行时状态完善。
