# Aneiang.Yarp 企业级开源治理路线图

> 目标：围绕身份一致性、存储迁移、性能、安全、项目结构、配置部署六大方向，把 Aneiang.Yarp 从功能型网关平台提升为企业级开源项目。  
> 原则：先止血、再过渡、再正式化；所有改造必须兼容现有 API 和现有 SQLite 数据。

---

## 0. 总体执行优先级

| 优先级 | 阶段 | 目标 | 状态 |
|---|---|---|---|
| P0 | S1 身份一致性止血 | 禁止普通编辑改 ID，重命名统一入口 | 已完成过渡版 |
| P0 | S2 存储迁移治理 | 统一迁移器、补齐缺字段、避免启动失败 | 已完成过渡版 |
| P0 | S3 写路径性能 | 增量持久化、异步快照、避免全量写 | 进行中 |
| P0 | S4 安全默认值 | 控制面认证、Dashboard 公网保护、gRPC 认证 | 待执行 |
| P1 | S5 项目结构分层 | Storage/Deployment/Plugin 拆分，收敛 public API | 待执行 |
| P1 | S6 配置部署标准化 | 顺序无关、自动中间件、热更新边界、部署模板 | 进行中 |

---

# 1. 身份与一致性必须项

## 1.1 目标

建立稳定的实体身份模型：

```text
UID：内部不可变身份
Key：YARP 对外 ID，可受控重命名
DisplayName：用户展示名，可自由修改
```

当前过渡字段：

```text
RouteUid + RouteId
ClusterUid + ClusterId
PolicyUid + PolicyId
```

最终命名建议：

```text
RouteUid + RouteKey + DisplayName
ClusterUid + ClusterKey + DisplayName
PolicyUid + PolicyKey + DisplayName
```

## 1.2 已完成

- [x] 普通编辑禁改 `routeId` / `clusterId`
- [x] 后端 Save API 拒绝通过普通保存改 ID
- [x] 独立重命名按钮和弹窗
- [x] `GatewayIdentityService` 统一处理 rename 后副作用
- [x] `RouteEntity` / `ClusterEntity` / `PolicyEntity` 增加 UID
- [x] `DynamicRouteConfig` / `DynamicClusterConfig` 增加 UID
- [x] `policy_targets` 关联表过渡版
- [x] `RouteEntity.ClusterUid` / `DynamicRouteConfig.ClusterUid` 过渡版
- [x] 重命名时保留 `RouteUid` / `ClusterUid`

## 1.3 必须继续做

### S1.1 正式区分 Key 与 ID

任务：

- [ ] 将内部语义从 `RouteId` 逐步改为 `RouteKey`
- [ ] 将内部语义从 `ClusterId` 逐步改为 `ClusterKey`
- [ ] 将内部语义从 `PolicyId` 逐步改为 `PolicyKey`
- [ ] 对外 DTO 保持旧字段兼容一个版本周期
- [ ] UI 上显示三层身份：`UID / Key / DisplayName`

验收：

- 内部关联不再依赖可变 Key
- YARP 输出仍使用 Key
- 旧 API 不破坏

### S1.2 策略绑定彻底切换到 `policy_targets`

任务：

- [x] 新增 `policy_targets`
- [x] Apply/Unapply/Delete/Get 读写 `policy_targets`
- [ ] 旧 `applied_targets` 改为只读兼容
- [ ] 后续版本停止写 `applied_targets`
- [ ] 最终删除 `applied_targets`

验收：

- 修改 Route/Cluster Key 后策略绑定不丢
- 策略绑定以 UID 为准
- UI 显示使用 `target_key_snapshot`

---

### S1.3 Rename 全事务化

任务：

- [x] `GatewayIdentityService` 过渡版
- [ ] Rename 主流程添加事务边界
- [ ] RenameCluster 覆盖 route 引用、policy target、circuit state、audit、history、notification
- [ ] RenameRoute 覆盖 policy target、audit、history、notification
- [ ] Rename 失败可回滚

验收：

- 任意 rename 不产生半更新状态
- rename 后当前配置一致
- 历史日志保留旧 key snapshot

---

### S1.4 运行时状态 UID 化

任务：

- [ ] 熔断状态 key 从 `clusterKey:destinationKey` 迁移为 `clusterUid:destinationUid`
- [ ] 保留 `clusterKeySnapshot` / `destinationKeySnapshot` 展示

# 2. 存储与迁移必须项

## 2.1 目标

从“仓储自建表 + 零散 ALTER”升级为企业级 migration 管理。

## 2.2 已完成

- [x] 新增 `SqliteSchemaMigrator`
- [x] 启动统一补齐新增字段
- [x] 自动回填 UID 与 snapshot 字段
- [x] `policy_targets` 自动建表
- [x] 避免仓储并发初始化导致缺字段

## 2.3 必须继续做

### S2.1 引入 migration version 表

新增表：

```sql
CREATE TABLE schema_migrations (
    id TEXT PRIMARY KEY,
    description TEXT NOT NULL,
    checksum TEXT NULL,
    applied_at TEXT NOT NULL
);
```

任务：

- [ ] 将当前迁移拆成版本化脚本
- [ ] 每个迁移有 ID，例如 `20260618_001_add_identity_uid`
- [ ] 迁移执行前检查是否已应用
- [ ] 记录 applied_at/checksum
- [ ] 迁移失败日志明确

验收：

- 启动日志能看到已应用/跳过迁移
- 老库升级新版本无缺字段异常
- 迁移可重复执行且无副作用

---

### S2.2 迁移事务化

任务：

- [ ] migration 执行使用事务边界
- [ ] DDL/DML 分阶段执行
- [ ] 大表回填分批执行
- [ ] 迁移超时和失败恢复策略

验收：

- 中途失败不会留下半迁移状态
- 大表升级不会卡死启动

---

### S2.3 仓储去迁移逻辑

任务：

- [ ] 仓储 `EnsureInitializedAsync` 只做轻量检查
- [ ] 建表/改表全部迁入 `SqliteSchemaMigrator`
- [ ] 避免每个仓储重复 `ALTER TABLE`

验收：

- 所有 schema 变更只在 migrator 中定义
- 仓储只负责 CRUD

---

### S2.4 拆分 Storage 项目

建议新增：


任务：

- [ ] 将存储实体/接口移动到 Abstractions

- [ ] Dashboard 只依赖抽象和具体实现包

# 3. 性能必须项

## 3.1 目标

写路径和代理热路径必须保持轻量，普通新增/编辑不能因全量快照和全量持久化变慢。

## 3.2 已完成

- [x] `AddOrUpdateRoute` 增量保存单 route
- [x] `UpdateRouteMetadata` 增量保存单 route
- [x] `AddCluster` / `UpdateCluster` / `CreateCluster` 增量保存单 cluster + destinations
- [x] `UpdateClusterCircuitBreaker` 增量保存单 cluster
- [x] 新增 `ConfigSnapshotScheduler` 后台快照队列
- [x] 普通保存 route/cluster 改为异步 after snapshot
- [x] 删除/重命名/导入/回滚保留同步 before snapshot
- [x] 迁移器避免每次启动全表更新 proxy_logs / notification_history

## 3.3 必须继续做

### S3.1 配置快照策略配置化

配置建议：

```json
"Gateway": {
  "Dashboard": {
    "ConfigHistory": {
      "AutoSnapshotBeforeMutation": true,
      "AsyncSnapshotForLowRiskMutation": true,
      "MaxSnapshots": 50
    }
  }
}
```

任务：

- [ ] 新增 `ConfigHistoryOptions`
- [ ] 普通保存按配置决定是否排队快照
- [ ] 高风险操作强制同步 before snapshot
- [ ] 快照队列可观测：队列长度、失败次数

验收：

- 普通保存响应时间不随配置规模线性增长
- 删除/重命名仍可回滚

---

### S3.2 代理热路径轻量化

任务：

- [ ] 请求/响应 body 捕获默认关闭或严格限制
- [ ] 大文件/流式响应/SSE 不进入 body buffer
- [ ] 日志采样默认可配置
- [ ] `Console.WriteLine` 全部替换为 `ILogger`

验收：

- 关闭日志时代理开销接近原生 YARP

任务：

- [ ] logs 分页/筛选

## 7. 建议执行顺序

1. **安全默认值 P0**：控制面认证、公网绑定保护、gRPC 认证
2. **迁移版本表 P0**：`schema_migrations`
3. **性能治理收尾 P0**：快照策略配置化、热路径日志限制
4. **Storage 拆分 P1**：抽象与 SQLite 实现分离
5. **UID 正式模型化 P1**：Key 命名、DTO 兼容、前端展示三层身份
6. **测试体系 P1**：迁移测试、rename 测试、性能 benchmark
7. **部署模板 P1**：多进程/Docker/systemd/Windows Service

---

## 8. 验收总标准

企业级开源达标前，必须满足：

- [ ] 生产环境控制面无认证无法启动
- [ ] 老 SQLite 库升级不缺字段、不丢数据
- [ ] route/cluster 重命名不影响策略绑定
- [ ] 普通写操作延迟不随全量配置线性增长
- [ ] 日志、审计、通知都保留 key snapshot
- [ ] Core 不依赖 Dashboard 存储实现
- [ ] 所有公开 API 有明确兼容承诺
- [ ] CI 覆盖 build/test/pack/vulnerability scan
- [ ] 有迁移测试和性能 benchmark
